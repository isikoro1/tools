# リファクタリング計画

## 対象

`matrix-rain-simulator` は、Canvasでマトリックス風の文字雨を生成できる静的Webツールです。

現在は以下の機能を持っています。

- 文字プリセット切り替え
- カスタム文字入力
- 文字サイズ、速度、密度、残像、発光、奥行きの調整
- PNG出力
- GIF出力
- MP4/WebM動画出力
- JSON設定保存・読込
- カスタムプリセット保存
- フルスクリーン表示

## 目的

このリファクタリングの目的は、既存機能を維持しつつ、コードの保守性、可読性、説明性を高めることです。

現状でもツールとしては動作していますが、主要な処理が `app.js` に集中しているため、今後の修正や機能追加が難しくなりやすい構成です。

本計画では、以下を重視します。

- 既存機能を壊さない
- 大きなUI変更を避ける
- 静的Webページとしての構成を維持する
- 外部依存を増やさない
- 公開向けの安全性方針を弱めない
- 責務を分離し、変更しやすい構造にする

## 現状の課題

### 1. `app.js` に責務が集中している

現在の `app.js` には、以下の処理がまとまっています。

- DOM要素の参照
- 設定値の取得・反映
- プリセット管理
- localStorage管理
- Canvas描画
- アニメーション状態管理
- PNG出力
- GIFエンコード
- 動画出力
- イベントリスナー登録

動作確認や小規模な修正には問題ありませんが、今後機能を増やす場合、影響範囲が読み取りにくくなります。

### 2. JSON読込時の入力値正規化を強化したい

JSON設定の読込時に、想定外の値が入っても画面や描画設定が壊れにくいように、値の正規化を強化します。

改善方針:

- 数値は有限値か確認する
- range inputは `min` / `max` の範囲に収める
- checkboxはbooleanとして安全に変換する
- selectは存在するoption値のみ許可する
- 不正な値は `defaultConfig` へフォールバックする

### 3. エクスポート処理と描画状態が密結合している

GIF出力時、フレーム生成のために本体シミュレーションを進める処理があります。

そのため、GIFを書き出すと画面上のアニメーション状態も進みます。現状でも仕様として許容できますが、理想的には表示用stateとエクスポート用stateを分離した方が保守しやすくなります。

### 4. READMEが不足している

ツールの概要、使い方、技術的な工夫点、今後の改善方針を説明するREADMEを追加します。

GitHub上でポートフォリオとして見せる場合、コードだけでなく「何を作ったのか」「何を工夫したのか」「今後どう改善するのか」が分かる状態にする必要があります。

## 改善方針

## 優先度A: 最優先で対応する

### 1. README.mdを追加する

`matrix-rain-simulator/README.md` を追加し、以下を記載します。

- 概要
- 主な機能
- 使用技術
- 使い方
- 工夫した点
- 今後の改善候補

READMEを追加することで、GitHub上でプロジェクトの意図と技術要素が伝わりやすくなります。

### 2. `app.js` の責務分割を段階的に進める

現在の単一ファイル構成を見直し、責務ごとにファイルを分割します。

想定構成:

```txt
matrix-rain-simulator/
  index.html
  styles.css
  README.md
  docs/
    prompt.md
    refactoring-plan.md
  src/
    main.js
    dom.js
    settings.js
    presets.js
    rain-engine.js
    renderer.js
    exporters/
      png.js
      gif.js
      video.js
    utils/
      color.js
      random.js
```

最初から完全分割が難しい場合は、以下の順に段階的に進めます。

1. DOM参照とイベント接続を分離
2. 設定値の取得・適用・正規化を分離
3. プリセット管理を分離
4. 描画・アニメーション状態管理を分離
5. PNG/GIF/動画出力を分離
6. 色変換・乱数などの補助関数を分離

### 3. JSON読込時のバリデーションを追加する

設定ファイル読込時に、値を安全に正規化します。

対応方針:

- numberは `Number.isFinite()` で検証する
- range inputは `min` / `max` の範囲に収める
- checkboxはbooleanとして安全に変換する
- selectは存在するoption値のみ許可する
- 不正な値は `defaultConfig` へフォールバックする

## 優先度B: できれば対応する

### 4. GIF出力と本体シミュレーション状態を分離する

GIF出力時に本体のアニメーション状態を直接進めないようにします。

改善案:

- エクスポート開始時にシミュレーション状態をcloneする
- export専用stateを使ってフレームを生成する
- 本体表示用stateには影響を与えない

すぐに完全対応しない場合でも、READMEの今後の改善候補に明記します。

### 5. 動画/GIF出力のFPSラベルを見直す

動画出力でもGIF用FPS設定を使用しているため、UI上の `GIF FPS` は誤解を招く可能性があります。

最小対応としては、`Export FPS` へのリネームで十分です。

### 6. `downloadBlob()` のObject URL解放を安全寄りにする

`URL.revokeObjectURL()` を `link.click()` の直後に実行すると、ブラウザによってはタイミングが早すぎる可能性があります。

改善案:

- 一時リンクを `document.body` に追加する
- `click()` 後にリンクを削除する
- `setTimeout()` でObject URLを解放する

## 優先度C: 余裕があれば対応する

### 7. 設定項目をメタデータ化する

`defaultConfig`、`settingKeys`、input要素の `min` / `max` / `option` が分散しているため、将来的には設定スキーマとして一元管理します。

この形にすると、UI反映、JSON読込、初期化、バリデーションを共通化しやすくなります。

### 8. テストしやすい純粋関数を切り出す

以下のようなDOMやCanvasに依存しない関数は、テストしやすい形で分離します。

- `clampNumber`
- `normalizeBoolean`
- `hexToRgb`
- `mix`
- `colorToCss`
- `distributionSample`
- `toFullWidth`
- `characterAt`

すぐにテストランナーを導入しなくても、純粋関数として分けるだけで将来のテスト導入がしやすくなります。

## 変更しない方針

以下は今回の改善対象外とします。

- フレームワークの導入
- 大規模なUIデザイン変更
- 主要機能の削除
- 外部CDNや外部script依存の追加
- CSPの緩和
- サーバーサイド化
- 認証機能の追加
- localStorageへの秘密情報保存

## 受け入れ条件

以下を満たせば、今回のリファクタリングは完了とします。

- ページが正常に表示される
- Matrix rain animation が動作する
- 各スライダー、select、color inputが動作する
- プリセット切り替えが動作する
- カスタムプリセット追加・更新・削除が動作する
- PNG出力が動作する
- GIF出力が動作する
- 動画出力が対応ブラウザで動作する
- JSON保存・読込が動作する
- 不正なJSON設定値で画面が壊れない
- README.md が追加されている
- `docs/prompt.md` に今回の変更概要が追記されている
- 既存のCSP方針が維持されている

## 最終ゴール

このツールを、単なる動くミニツールではなく、以下が伝わるポートフォリオにすることを目指します。

- Canvas APIを使った描画実装ができる
- Vanilla JavaScriptで状態管理とUI操作を実装できる
- ブラウザAPIを使って画像・動画・JSON出力ができる
- CSPやlocalStorageの扱いに一定の公開向け配慮がある
- 既存コードをレビューし、保守性改善の計画を立てられる
- 計画に沿って段階的にリファクタリングできる
